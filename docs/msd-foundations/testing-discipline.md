# Testing Discipline — The Foundation for AI-Generated Code

!!! quote "Dave Farley, *Modern Software Engineering*"
    "Tests are the only reliable specification of software behaviour. Without them, you cannot
    know whether a change — from any source — has broken anything."

!!! quote "Martin Fowler, *Refactoring*"
    "Whenever you are tempted to type something into a print statement or a debugger expression,
    write it as a test instead."

---

## Why Testing Discipline Matters More with AI

Testing has always been important. With AI-generated code, it becomes non-negotiable.

The fundamental problem with LLM output is that it is *optimised for plausibility, not correctness*.
An LLM produces code that reads like code written by a competent team member. The formatting is clean,
the variable names are sensible, the logic follows a recognisable structure. And it can be subtly,
silently wrong in ways that a skilled reviewer will miss on first reading.

For the O&A team, the risk is concrete:

- A NETCONF `edit-config` operation that uses `merge` instead of `replace` on an interface
  configuration list will appear to work but leave stale entries on the device.
- A YANG `must` constraint generated by an LLM may reference the wrong relative path, allowing
  invalid configuration to pass schema validation.
- An Ansible task that the LLM marks as `changed_when: false` to silence idempotency noise
  silently masks real failures.
- A Python function that wraps a TMF API client may swallow HTTP 4xx responses and return an
  empty result instead of raising an exception.

None of these are hypothetical. They are patterns that appear in AI-generated telecom automation
code. They are caught by tests. They are missed by code review alone.

**Strong testing discipline is the safety net that makes AI-assisted development safe.**

## The Testing Pyramid for O&A

Martin Fowler's testing pyramid defines three layers of tests, distinguished by speed, scope, and
cost. For the O&A team, each layer has specific technologies and concerns:

```
                    ┌─────────────────────┐
                    │   End-to-End (E2E)  │  Few, slow, high-confidence
                    │  Network Scenario   │  Full NSD provisioning flows
                    └─────────────────────┘
               ┌─────────────────────────────────┐
               │         Integration             │  Medium count, medium speed
               │  NETCONF sessions, API contracts │  Real netsim, contract mocks
               └─────────────────────────────────┘
          ┌─────────────────────────────────────────────┐
          │                   Unit                      │  Many, fast, isolated
          │  Python/Go functions, YANG model validation  │  Pure logic, no I/O
          └─────────────────────────────────────────────┘
```

### Unit Tests — The Base of the Pyramid

Unit tests cover isolated logic with no external dependencies. They run in milliseconds. There
should be hundreds of them. They are the first thing an AI-generated change must pass.

**O&A unit test scope:**

| Component | What to Unit Test | Tooling |
|---|---|---|
| Python network client | Request construction, response parsing, error handling | `pytest`, `unittest.mock` |
| Go NETCONF library | RPC builder, namespace handling, error mapping | `go test`, `testify` |
| YANG model validation | Individual leaf constraints, `must` expressions, `pattern` regexes | `pyang` in test mode, `yanglint` |
| Ansible filter plugins | Data transformation, IP normalisation, VLAN mapping | `pytest` against plugin functions |
| TMF API request builders | Payload construction against schema, field validation | `pytest`, `jsonschema` |
| NSO Python service code | Template rendering, plan item logic, self-test hooks | `pytest` with NSO Python API mocks |

**Example — unit testing a YANG `must` constraint:**

```python
# tests/unit/test_yang_constraints.py
import subprocess
import pytest

YANG_MODEL = """
module noa-interface {
  namespace "urn:noa:interface";
  prefix noa-if;

  container interface {
    leaf bandwidth-mbps {
      type uint32;
      must ". <= 100000" {
        error-message "Bandwidth must not exceed 100 Gbps";
      }
    }
    leaf description {
      type string {
        length "1..255";
      }
    }
  }
}
"""

@pytest.mark.parametrize("bandwidth,should_pass", [
    (1000, True),
    (100000, True),       # boundary — valid
    (100001, False),      # boundary + 1 — must constraint violation
    (0, True),
    (999999, False),
])
def test_bandwidth_must_constraint(bandwidth: int, should_pass: bool, tmp_path):
    """YANG must constraint on bandwidth is enforced correctly."""
    model_file = tmp_path / "noa-interface.yang"
    model_file.write_text(YANG_MODEL)

    # Generate instance data and validate with yanglint
    instance = f"""<interface><bandwidth-mbps>{bandwidth}</bandwidth-mbps></interface>"""
    instance_file = tmp_path / "instance.xml"
    instance_file.write_text(instance)

    result = subprocess.run(
        ["yanglint", "-t", "data", str(model_file), str(instance_file)],
        capture_output=True,
        text=True,
    )
    passed = result.returncode == 0
    assert passed == should_pass, (
        f"Bandwidth={bandwidth}: expected {'pass' if should_pass else 'fail'}, "
        f"got {'pass' if passed else 'fail'}. stderr: {result.stderr}"
    )
```

### Integration Tests — The Middle Layer

Integration tests verify that components work together correctly. They require real or simulated
infrastructure. They run in seconds to minutes. The O&A team should have tens to low hundreds.

**O&A integration test scope:**

| Scope | What to Test | Tooling |
|---|---|---|
| NETCONF session | Open session, get-config, edit-config, validate, commit — against `netsim` | `ncclient`, `pytest` |
| NSO service provisioning | Deploy a service, verify device config, rollback — against NSO `netsim` | NSO Python test framework |
| TMF API contract | POST to Resource Inventory, verify response schema matches TMF639 | `pytest`, `schemathesis` |
| Ansible playbook execution | Run playbook against `molecule` test environment, assert target state | `molecule`, `testinfra` |
| Python service API | POST a provisioning request, verify downstream NETCONF RPC shape | `pytest`, `httpx`, `respx` |

**Example — NETCONF edit-config integration test:**

```python
# tests/integration/test_netconf_interface.py
import pytest
from ncclient import manager
from ncclient.xml_ import to_ele

NETSIM_HOST = "localhost"
NETSIM_PORT = 12022
NETSIM_USER = "admin"
NETSIM_PASS = "admin"

INTERFACE_CONFIG = """
<config>
  <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
    <interface>
      <name>GigabitEthernet0/0/0</name>
      <description>AI-provisioned uplink</description>
      <type xmlns:ianaift="urn:ietf:params:xml:ns:yang:iana-if-type">
        ianaift:ethernetCsmacd
      </type>
      <enabled>true</enabled>
    </interface>
  </interfaces>
</config>
"""

@pytest.fixture(scope="module")
def netconf_session():
    with manager.connect(
        host=NETSIM_HOST,
        port=NETSIM_PORT,
        username=NETSIM_USER,
        password=NETSIM_PASS,
        hostkey_verify=False,
    ) as m:
        yield m

def test_edit_config_creates_interface(netconf_session):
    """edit-config with merge operation creates the interface on netsim."""
    result = netconf_session.edit_config(
        target="candidate",
        config=INTERFACE_CONFIG,
    )
    assert result.ok

    netconf_session.commit()

    # Verify via get-config
    filter_xml = to_ele("""
        <filter type="subtree">
          <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
            <interface><name>GigabitEthernet0/0/0</name></interface>
          </interfaces>
        </filter>
    """)
    get_result = netconf_session.get_config(source="running", filter=filter_xml)
    assert "AI-provisioned uplink" in str(get_result)

def test_edit_config_is_idempotent(netconf_session):
    """Applying the same interface config twice produces no error and no duplicate."""
    netconf_session.edit_config(target="candidate", config=INTERFACE_CONFIG)
    netconf_session.commit()

    netconf_session.edit_config(target="candidate", config=INTERFACE_CONFIG)
    netconf_session.commit()

    filter_xml = to_ele("""
        <filter type="subtree">
          <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces" />
        </filter>
    """)
    result = netconf_session.get_config(source="running", filter=filter_xml)
    # Count occurrences — should be exactly one
    interface_count = str(result).count("GigabitEthernet0/0/0")
    assert interface_count == 1, f"Expected 1 interface entry, found {interface_count}"
```

### End-to-End Tests — The Apex

E2E tests exercise complete business flows through the system as deployed. They are slow (minutes),
expensive to maintain, and should be few in number. They exist to give the team confidence that
the system works as a whole, not to cover every code path.

**O&A E2E test scenarios:**

| Scenario | Flow | Pass Criteria |
|---|---|---|
| Full NSD provisioning | API request → O&A service → NSO → netsim → device config | Device shows correct running config; provisioning record in inventory |
| Service teardown and rollback | Provision → delete → verify clean state | No residual config on simulated device |
| TMF fulfilment order processing | POST TMF641 order → orchestration → provisioning | TMF641 order reaches `completed` state; Resource Inventory updated |
| Config drift detection | Manually corrupt netsim config → run compliance check | Drift detected, report generated, remediation RPC issued |

## TDD as the Safety Net for AI-Generated Code

Test-Driven Development is not primarily a design technique when applied to AI-assisted development.
It is a *safety protocol*.

The TDD cycle in the AI-assisted O&A context:

```
1. TEAM MEMBER writes a failing test that specifies the desired behaviour.
2. TEAM MEMBER uses AI to generate code that makes the test pass.
3. TEAM MEMBER reviews the AI-generated code against the test's intent.
4. If the test passes and the code is coherent, merge.
5. AI can also suggest additional test cases — team member reviews before adding.
```

This workflow ensures two things:

- The test was written by a human who understood the requirement — not generated by the same model
  that generated the implementation.
- The AI's output is constrained by a specification it cannot rewrite.

**The anti-pattern** is inverting this: ask AI to write the implementation, then ask AI to write
tests for the implementation it just wrote. The AI will write tests that pass its own code. This
produces high test coverage numbers and zero safety net value.

### TDD Example — NETCONF RPC Builder

```python
# Step 1: Team Member writes the failing test FIRST
# tests/unit/test_rpc_builder.py

def test_edit_config_rpc_uses_replace_operation_for_lists():
    """
    When configuring a list resource, the RPC must use nc:operation='replace'
    to avoid stale entries. 'merge' is incorrect for list config.
    """
    builder = NetconfRpcBuilder()
    rpc_xml = builder.build_interface_list_config(
        interfaces=[
            {"name": "Gi0/0/0", "description": "uplink", "enabled": True},
            {"name": "Gi0/0/1", "description": "downlink", "enabled": False},
        ]
    )
    root = etree.fromstring(rpc_xml.encode())
    # The interfaces container must have operation="replace" — not "merge"
    ns = {"nc": "urn:ietf:params:xml:ns:netconf:base:1.0"}
    interfaces_elem = root.find(".//interfaces", namespaces=ns)
    assert interfaces_elem is not None
    assert interfaces_elem.get("{urn:ietf:params:xml:ns:netconf:base:1.0}operation") == "replace"


# Step 2: Team Member uses AI with this prompt:
# "Write a NetconfRpcBuilder class with a build_interface_list_config method.
#  The method takes a list of interface dicts and produces NETCONF edit-config XML.
#  The interfaces container MUST use nc:operation='replace' — not 'merge' — to
#  avoid stale entries when the list is reconfigured. Here is the failing test: [paste test]"

# Step 3: AI generates the implementation. Test is run. If it passes, review and merge.
```

## AI's Role in Testing

AI is genuinely useful in testing workflows when applied correctly:

### Generating Test Cases from Specifications

Given a YANG model, OpenAPI spec, or a written requirement, AI can generate a first set of test
cases covering nominal, boundary, and error scenarios. The team member reviews for completeness and
correctness — AI covers the breadth, team member covers the depth.

```
Prompt: "Given this YANG module [paste], generate pytest test cases for yanglint validation
covering: (1) all mandatory leaves present, (2) each mandatory leaf missing in isolation,
(3) each 'must' constraint violated, (4) 'pattern' regex boundaries, (5) 'range' boundaries.
Output as pytest.mark.parametrize decorators."
```

### Expanding Coverage for Existing Code

```
Prompt: "Here is our Python NETCONF session wrapper [paste code] and the existing tests
[paste tests]. Identify any behaviour paths not covered by the tests and generate
additional pytest test cases. Flag any case where the existing tests might be testing
the wrong thing — e.g., where error conditions are silently swallowed."
```

### Generating YANG Test Fixtures

```
Prompt: "Generate five XML test fixture files conforming to this YANG model [paste].
Include: one fully valid instance, one with a missing mandatory leaf, one with a
'must' constraint violation, one with an out-of-range value, and one with an invalid
'pattern' match. Each file should be a complete NETCONF get-config response body."
```

### Anti-Pattern: Trusting AI Test Output Without Review

!!! danger "Anti-Pattern — AI Tests That Validate Nothing"
    A common failure mode: an team member asks AI to write tests for existing code. The AI generates
    tests that call the code and assert `assert result is not None`. These tests pass every time,
    report 100% coverage for the tested function, and catch nothing.

    Always review AI-generated tests for:

    - **Specificity** — does the assertion verify the actual expected value, or just that something returned?
    - **Negative cases** — does the test cover what happens when the input is wrong?
    - **Isolation** — does the test use real I/O when it should mock, or mock when it should use real I/O?
    - **Idempotency** — if the test modifies state (e.g., writes to netsim), does it clean up?

## O&A-Specific Testing Examples

### Testing Ansible Playbooks with Molecule

```yaml
# molecule/default/verify.yml
---
- name: Verify interface configuration
  hosts: network_devices
  gather_facts: false
  tasks:
    - name: Get interface config via NETCONF
      community.yang.fetch:
        file: "{{ playbook_dir }}/yang/ietf-interfaces.yang"
        search_path: "{{ playbook_dir }}/yang"
      register: interface_config

    - name: Assert GigabitEthernet0/0/0 is configured
      assert:
        that:
          - interface_config.get_config.interfaces.interface |
            selectattr('name', 'equalto', 'GigabitEthernet0/0/0') |
            list | length == 1
        fail_msg: "Interface GigabitEthernet0/0/0 not found in running config"

    - name: Assert idempotency — no changes on second run
      community.yang.configure:
        file: "{{ playbook_dir }}/yang/ietf-interfaces.yang"
        # ... same config as provisioning play
      register: second_run_result

    - name: Verify no changes reported
      assert:
        that: not second_run_result.changed
        fail_msg: "Playbook is not idempotent — changes reported on second run"
```

### Testing TMF API Clients

```python
# tests/unit/test_tmf639_client.py
import pytest
import respx
import httpx
from noa.clients.tmf639 import ResourceInventoryClient

@pytest.fixture
def tmf_client():
    return ResourceInventoryClient(base_url="http://test-inventory.noa.local")

@respx.mock
def test_get_resource_returns_none_for_404(tmf_client):
    """Client returns None (not raises) when resource is not found."""
    respx.get("http://test-inventory.noa.local/resourceInventoryManagement/v4/resource/not-exist") \
        .mock(return_value=httpx.Response(404, json={"code": "404", "reason": "Not found"}))

    result = tmf_client.get_resource("not-exist")
    assert result is None  # Must not raise; callers handle missing resources

@respx.mock
def test_get_resource_raises_on_500(tmf_client):
    """Client raises an exception for 5xx responses — do not swallow server errors."""
    respx.get("http://test-inventory.noa.local/resourceInventoryManagement/v4/resource/r-001") \
        .mock(return_value=httpx.Response(500, json={"code": "500", "reason": "Internal error"}))

    with pytest.raises(ResourceInventoryClient.ServerError):
        tmf_client.get_resource("r-001")

@respx.mock
def test_create_resource_validates_response_schema(tmf_client):
    """create_resource validates the response body against TMF639 schema."""
    valid_response = {
        "id": "r-001",
        "href": "/resourceInventoryManagement/v4/resource/r-001",
        "name": "Router-A",
        "@type": "LogicalResource",
    }
    respx.post("http://test-inventory.noa.local/resourceInventoryManagement/v4/resource") \
        .mock(return_value=httpx.Response(201, json=valid_response))

    resource = tmf_client.create_resource({"name": "Router-A", "@type": "LogicalResource"})
    assert resource["id"] == "r-001"
    assert resource["@type"] == "LogicalResource"
```

## Coverage Targets and Tracking

Coverage numbers are a lagging indicator, not a goal in themselves. A suite with 95% coverage
that tests only happy paths provides less safety than one with 75% coverage that thoroughly tests
error paths and boundary conditions.

Recommended targets for the O&A team:

| Layer | Target Coverage | Priority Areas |
|---|---|---|
| Unit — Python/Go business logic | ≥ 80% line coverage | Error handling paths, boundary conditions, YANG constraint logic |
| Unit — YANG models | 100% `must` / `pattern` constraints tested | Every constraint must have at least one passing and one failing test case |
| Integration — NETCONF operations | ≥ 1 test per RPC type (get, edit-config, copy-config) | Idempotency, error responses, schema validation |
| Integration — TMF API clients | All 4xx and 5xx response codes handled | Client must not swallow errors silently |
| E2E — Provisioning flows | ≥ 1 test per service type in the catalogue | Full flow with verification against netsim |

**Tracking in CI:**

```yaml
# .gitlab-ci.yml — coverage reporting
unit-tests:
  stage: commit
  script:
    - pytest tests/unit/ \
        --cov=noa \
        --cov-report=xml:coverage.xml \
        --cov-fail-under=80
  coverage: '/TOTAL.*\s+(\d+%)$/'
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage.xml
```

## Checklist: Testing Discipline for AI-Assisted Development

### Test Suite Health

- [ ] Unit tests run on every commit and block merge on failure.
- [ ] Every YANG `must` constraint has at least one passing and one failing test case.
- [ ] Every Ansible playbook has a Molecule idempotency test.
- [ ] Every TMF API client method has a test for 4xx and 5xx responses.
- [ ] NETCONF operations are tested against `netsim` in the acceptance stage.

### TDD and AI Integration

- [ ] Tests are written before or alongside AI-generated code, not after.
- [ ] AI-generated test cases are reviewed by a human before merging.
- [ ] AI-generated tests are checked for specificity (not just `assert result is not None`).
- [ ] The same AI session is not used to generate both code and its tests.

### Coverage and Tracking

- [ ] Coverage is measured and reported in CI.
- [ ] Coverage targets are defined per layer and enforced with `--cov-fail-under`.
- [ ] Coverage reports are reviewed in sprint retrospectives, not just in PRs.
- [ ] Coverage gaps in YANG constraint testing are treated as high-priority issues.
