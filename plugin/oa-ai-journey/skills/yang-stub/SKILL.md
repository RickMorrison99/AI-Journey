# Skill: Generate YANG Stub

**Skill name**: `yang-stub`
**Trigger phrases**: "generate a YANG module", "create YANG stub", "new YANG model", "yang stub for"

---

## What This Skill Does

Generates a YANG 1.1 module stub following O&A conventions. The stub includes all mandatory structural elements, placeholder groupings, a revision block, and inline comments that guide the engineer to fill in real constraints. The output is valid, parseable YANG 1.1 that passes `pyang --lint` before any content is added.

---

## Spec-First Gate

Before generating any YANG, this skill asks:

> "Is there an existing YANG RFC (ietf-*, iana-*, openconfig-*) or vendor-supplied YANG model (Cisco-IOS-XR, junos-*, Nokia-*) that you should **augment** or **deviate** rather than create from scratch?"

If yes → do not generate a new module. Generate an `augment` extension stub targeting the existing model.
If no → proceed with the O&A stub generation.

Also ask:
- "What is the purpose of this module — service model, device model, or platform utility?"
- "What target device type and NOS vendor? (IOS-XR / JunOS / EOS / Nokia SR OS / vendor-agnostic)"
- "Is there a relevant TM Forum domain? (resource inventory, service ordering, performance, etc.)"

---

## O&A YANG Conventions

| Convention | Rule |
|-----------|------|
| Module prefix | Always `oa-` (e.g., `oa-vpn`, `oa-interface-policy`) |
| Namespace | Always `urn:oa:yang:[module-name]` |
| Revision | Mandatory, with `description` explaining what changed |
| Leaf grouping | All leaf definitions go in `grouping` blocks — never inline in `container` |
| Constraints | Use `must` expressions for constraints; prefer `must` over `when` where both would work |
| Description | Every `container`, `list`, `leaf`, and `grouping` requires a `description` statement |
| Reference | Include `reference` statement citing the RFC or TM Forum spec where applicable |
| Key leaves | All `list` keys must be `type string` or `type leafref` — never derived types |
| NSO service | Service model must `uses ncs:service-data` and `ncs:servicepoint` |

---

## Output Format

A complete, valid YANG 1.1 skeleton. It passes `pyang --lint` with no errors. Placeholder comments marked `// TODO:` guide the engineer to fill in real values.

## Example: NSO Service Model Stub

**Trigger**: "yang stub for a VPN L3 service"

```yang
module oa-l3vpn {
  yang-version 1.1;
  namespace "urn:oa:yang:oa-l3vpn";
  prefix oa-l3vpn;

  import ietf-inet-types {
    prefix inet;
  }
  import tailf-ncs {
    prefix ncs;
  }
  import tailf-common {
    prefix tailf;
  }

  organization "O&A Platform Team";
  contact "oa-platform@example.com";
  description
    "O&A L3VPN service model stub.
     Reference: TMF641 Service Ordering, RFC 4364 BGP/MPLS IP VPNs.
     TODO: Replace stub descriptions with production documentation.";

  revision 2025-01-01 {
    description
      "Initial stub — generated by oa-ai-journey yang-stub skill.
       TODO: Update revision date and description on first real change.";
    reference "https://github.com/RickMorrison99/AI-Journey";
  }

  // -------------------------------------------------------
  // Groupings — define all leaf sets here before containers
  // -------------------------------------------------------

  grouping l3vpn-endpoint-config {
    description "Configuration parameters for a single L3VPN endpoint.";

    leaf device {
      type leafref {
        path "/ncs:devices/ncs:device/ncs:name";
      }
      mandatory true;
      description "Target managed device for this endpoint.";
    }

    leaf interface {
      type string;
      mandatory true;
      description
        "Interface name on the device (e.g., GigabitEthernet0/0/0/1).
         TODO: Add must expression validating interface exists on device.";
    }

    leaf vrf-name {
      type string {
        length "1..64";
        pattern '[A-Za-z0-9_\-]+';
      }
      description "VRF name. Must be unique per device.";
    }

    leaf route-distinguisher {
      type string;
      description
        "Route distinguisher in ASN:nn or IP-address:nn format.
         TODO: Add must expression enforcing RD format.";
    }
  }

  grouping l3vpn-service-config {
    description "Top-level L3VPN service configuration.";

    leaf vpn-id {
      type string {
        length "1..32";
      }
      description "Unique VPN service identifier.";
    }

    list endpoint {
      key "device interface";
      min-elements 2;
      description "L3VPN service endpoints. Minimum two endpoints required.";

      must "count(../endpoint) >= 2" {
        error-message "An L3VPN service requires at least two endpoints.";
      }

      uses l3vpn-endpoint-config;
    }
  }

  // -------------------------------------------------------
  // Service entry point
  // -------------------------------------------------------

  list l3vpn {
    description "L3VPN service instances.";

    uses ncs:service-data;
    ncs:servicepoint "oa-l3vpn-servicepoint";

    key "vpn-id";

    uses l3vpn-service-config;
  }
}
```

## Post-Generation Checklist

After generating the stub, always remind the engineer:

1. Run `pyang --lint oa-[module-name].yang` — fix any warnings before committing
2. Commit the YANG module to `packages/[service-name]/src/yang/`
3. Draft an ADR if this is a new service model (use the `oa-adr` skill)
4. Check TMF domain relevance — if this is a resource or service model, reference the TMF Open API in the module description
